Проверь и приведи проект к FSD-архитектуре. Не ограничивайся анализом: если видишь нарушения, сразу сделай рефакторинг и доведи до проходящих проверок.

Что проверить и исправить:

1. FSD-структура
- `app` должен содержать только роутинг, layout/page-обертки и подключение screen/layout.
- `screens/pages` должны быть тонкими: без бизнес-логики, без больших данных, без сложной верстки, только композиция виджетов/features/entities/shared.
- Крупные самостоятельные блоки страницы выноси в `widgets`.
- Пользовательские сущности, компании, профиль и подобные доменные части выноси в `entities`, если они переиспользуемые.
- Действия и сценарии пользователя выноси в `features`.
- Базовые UI-компоненты, утилиты, tokens, config выноси в `shared`.

2. Папки внутри слайсов
- Компоненты и стили: `ui`.
- Данные/константы для отображения: `data`.
- Непропсовые типы: `model/types.ts`.
- Хуки, состояние, адаптеры состояния: `model`.
- Чистые helper-функции: `lib`.
- Props-типы, которые относятся только к одному компоненту, можно оставлять рядом с компонентом в `ui`.

3. Хуки, helpers, data, types
- Найди хуки внутри `ui` и вынеси их в `model`, если это не совсем локальная мелочь компонента.
- Найди чистые функции форматирования, маппинга, нормализации, масок, вычислений и вынеси их в `lib`.
- Найди массивы, mock-данные, тексты, пункты меню, tabs, options, config для UI и вынеси их в `data`.
- Найди доменные типы и типы данных, которые не являются props конкретного компонента, и вынеси их в `model/types.ts`.
- Проверь public API/imports, чтобы импорты шли через корректные слои и не нарушали FSD-зависимости.

4. Layout
- Если есть общий layout для раздела, вынеси его в `widgets/layout/<section>`.
- Внутри layout отдельно разложи `header`, `footer`, `sidebar`, `tabs`, `navigation` и другие части, если они достаточно самостоятельные.
- Учти, что потом могут появляться новые разделы, поэтому layout должен быть расширяемым.

5. Переиспользование компонентов
- Используй готовые компоненты проекта: `Typography`, `Button`, `Input`, `Card`, `Checkbox`, `Modal`, `Select` и другие.
- Не дублируй кнопки, поля, карточки, аватары, икон-кнопки и типографику вручную, если уже есть shared/base компонент.
- Если повторяется одинаковый UI-блок, вынеси его в отдельный компонент в правильный слой.

6. Типографика и размеры
- Размер шрифта должен задаваться через `Typography`, а не inline-стилями и не случайными CSS-классами.
- Не задавай `font-size` локально, если это должен контролировать `Typography`.
- Размеры в стилях задавай в `rem`. Если в макете размер в px, переведи его в rem, например `48px = 3rem` при базовых 16px.
- Не используй inline styles для обычной верстки. Стили должны быть вынесены в отдельные CSS/SCSS module файлы.

7. Цвета и дизайн-токены
- Не используй hardcoded цвета в компонентах и module-стилях: hex, rgb, rgba, hsl.
- Цвета должны идти из CSS variables/design tokens проекта.
- Исключение допустимо только в файлах глобальных токенов/темы, где эти переменные объявляются.
- Если нужного токена нет, добавь его в подходящее место глобальных стилей/темы и используй через `var(...)`.

8. Стили
- У каждого сложного компонента стили должны быть отдельно в `styles.module.css` или существующем принятом формате проекта.
- Не смешивай стили разных виджетов/фич в одном файле.
- Не оставляй большие style-объекты внутри TSX.
- Удали неиспользуемые классы и пустые файлы/папки.

9. Рефакторинг
- Если компонент слишком большой, раздели его на небольшие компоненты по смыслу.
- Если screen содержит верстку целого раздела, вынеси ее в widget.
- Если feature содержит UI и бизнес-логику вперемешку, раздели на `ui`, `model`, `lib`, `data`.
- Не делай лишних абстракций, но все повторяющееся и потенциально переиспользуемое вынеси.

10. Проверки
После изменений обязательно запусти:
- `pnpm typecheck`
- `pnpm lint`
- `pnpm lint:fsd`
- дополнительно просканируй проект через `rg` на:
  - хуки внутри `ui`
  - helper-функции внутри компонентов
  - hardcoded цвета
  - размеры в `px`
  - inline styles
  - данные/массивы/options внутри UI
  - пустые папки
